Preparación técnica basada en tus 21 proyectos del portfolio + mejores prácticas y seguridad a nivel auditor.

1 · Fundamentos de Solidity & EVM

Data location: storage / memory / calldata

El concepto que SIEMPRE preguntan
  • storage: persistente, vive on-chain en slots de 32 bytes. Caro (escribir = ~20k gas la primera vez). Las state variables viven acá.
  • memory: temporal, existe durante la ejecución de la función. Más barata. Mutable.
  • calldata: read-only, los argumentos de funciones external. La más barata (no se copia). Inmutable.
Regla práctica: parámetros de arrays/structs/strings en funciones external → calldata (ahorra gas vs memory).

Value types vs reference types

Value (uint, address, bool, bytes32): se copian por valor. Reference (arrays, structs, mappings, strings): apuntan a una ubicación; cuidado con storage pointers (modifican el original) vs copias en memory.

Storage layout (slots)

El estado se guarda en slots de 32 bytes, en orden de declaración. Variables chicas consecutivas se empaquetan en un mismo slot (ej. dos uint128, o uint96 + address). Los mapping y arrays dinámicos usan keccak256 del slot para ubicar sus datos. Clave para gas (packing) y para entender proxies (storage collision).

Visibilidad & state mutability

VisibilidadQuién llama
externalSolo desde afuera. Args en calldata → más barato.
publicAfuera + adentro. Genera getter automático en state vars.
internalEste contrato + los que heredan.
privateSolo este contrato (igual es visible on-chain, no es secreto).

Mutability: view (lee estado, no modifica), pure (ni lee ni modifica), payable (puede recibir ETH). Llamar una view/pure desde afuera es gratis (eth_call); desde otra función on-chain igual cuesta gas.

msg.sender vs tx.origin

Pregunta clásica de seguridad msg.sender = el caller directo (puede ser un contrato). tx.origin = el EOA que originó toda la transacción. Nunca uses tx.origin para autorización: un contrato malicioso intermedio puede hacer que tu check pase (phishing). Auth siempre con msg.sender. → Proyecto 03.

constant vs immutable

constant: fijo en compile-time, inlineado en el bytecode (gratis de leer). immutable: se setea una vez en el constructor y queda fijo; ideal para direcciones de tokens/feeds. Ambos ahorran gas vs storage normal.

Errores: require / revert / custom errors

require(cond, "msg");            // legible, string cuesta gas
if (!cond) revert MyError(x);    // custom error: ~4 bytes selector, más barato
revert("msg");                   // revert directo

Desde 0.8.4 los custom errors son el estándar: más baratos en gas y testeables por selector (vm.expectRevert(MyError.selector)). Un revert deshace TODO el estado de la tx (atomicidad).

fallback() y receive()

receive() external payable se ejecuta cuando llega ETH con calldata vacía. fallback() cuando no matchea ninguna función (o no hay receive). Mantenelas mínimas.

Cómo viaja una transacción

EOA firma → mempool → validador la incluye en un bloque → la EVM ejecuta el bytecode → cambios de estado + logs (events) + gas consumido. Si revierte, se deshace todo menos el gas gastado hasta ahí.

2 · Seguridad — nivel auditor

Estructurado según el OWASP Smart Contract Top 10 (2025). En 2025 se perdieron +US$2.3 billones por exploits; access control y reentrancy lideran. La mentalidad de auditor: ante cada función preguntate "¿cómo la rompo / cómo dreno fondos?".

Defensa en capas (lo que un auditor quiere oír) Código seguro desde el diseño → herramientas automáticas (Slither, fuzzing) → auditoría externa → bug bounty. Ninguna capa sola alcanza.

SC01 · Access Control (la #1 en pérdidas)

Funciones sensibles sin protección o mal protegidas. Prevención: onlyOwner/roles (Ownable, AccessControl), msg.sender (no tx.origin), validar quién puede llamar funciones init/admin. → Proyectos 04, 06, 15, 16.

Bug real que encontraste (Proyecto 16) canBorrow tenía un early-return (if (ratio == max) return true) que hacía que el primer préstamo (sin deuda previa) nunca se validara contra el colateral → cualquiera drenaba el pool. Fix: sacar el atajo y simular siempre la deuda nueva. Lección: un atajo válido en un contexto (canWithdraw) es un agujero en otro (canBorrow).

SC02 · Price Oracle Manipulation

Atacante distorsiona el precio que lee el contrato (típicamente con un flash loan sobre un pool de baja liquidez usado como oráculo). Prevención: usar oráculos robustos (Chainlink), nunca el spot price de un DEX como fuente directa; usar TWAP; validar freshness (staleness check). → Proyectos 11, 16.

Definiciones — TWAP y staleness
  • TWAP (Time-Weighted Average Price): precio promedio ponderado por tiempo en una ventana (ej. últimos 30 min). El caso típico es el oráculo de Uniswap. Es resistente a manipulación: para mover un TWAP de 30 min habría que sostener el precio falso 30 min (carísimo), no alcanza un flash loan de 1 bloque. Más seguro que el "spot price" instantáneo.
  • Staleness (precio rancio): Chainlink actualiza cada cierto tiempo (su "heartbeat") o al moverse X%. Si el feed se atrasa/rompe, latestRoundData() devuelve un precio viejo. Usarlo como actual = liquidar/prestar mal. El check valida que el dato sea reciente.
(, int256 price,, uint256 updatedAt,) = feed.latestRoundData();
require(price > 0, "bad price");                      // precio válido
require(block.timestamp - updatedAt < MAX_DELAY, "stale"); // freshness
(, int256 price,, uint256 updatedAt,) = feed.latestRoundData();
require(price > 0, "bad price");
require(block.timestamp - updatedAt < MAX_DELAY, "stale price");

SC03 · Logic Errors

Errores en la lógica de negocio: contabilidad mal hecha, redondeos, decimales. Prevención: tests exhaustivos (incluyendo fuzz/invariant), valuar todo en una unidad común.

Bug real (Proyecto 16 — liquidate) collateralToSeize = amount × 1.05 trataba 1 unidad de deuda = 1 de colateral, sin convertir por precio ni decimales. El liquidador recibía cantidades absurdas. Fix: deuda repagada → USD → + penalidad → cantidad de colateral al precio actual. Lección general DeFi: nunca compares cantidades crudas de tokens con decimales/precios distintos; normalizá todo a USD con escala fija (18).

SC04 · Lack of Input Validation

No validar argumentos (address(0), montos 0, arrays de distinto length). Prevención: checks al inicio (fail fast).

SC05 · Reentrancy (la canónica)

Cómo funciona Una llamada externa (.call{value:}, safeTransfer a un contrato, callback de NFT) devuelve el control al atacante antes de que actualices el estado → el atacante re-entra y repite la operación con estado "viejo" (ej. drenar el balance).
Defensas (decilas en este orden)
  1. CEI (Checks-Effects-Interactions): validá → actualizá estado → recién ahí llamás afuera. Gratis y resuelve la mayoría.
  2. ReentrancyGuard (nonReentrant): mutex, segunda capa.
  3. Pull over push: que el usuario retire (claim) en vez de empujarle fondos.
→ Proyectos 12 (PoC del ataque + fix), 04, 08, 18.

SC06 · Unchecked External Calls

No chequear el valor de retorno de .call o de transfer/transferFrom (algunos tokens como USDT devuelven false en vez de revertir). Prevención: chequear el bool success del .call; usar SafeERC20 para tokens. → default del curso en todos los proyectos con tokens.

SC07 · Flash Loan Attacks

Préstamo sin colateral que se pide y devuelve en la misma tx. Por sí solo no es el bug — es el amplificador: se combina con oracle manipulation o lógica débil para mover precios/votos con capital "infinito". Prevención: oráculos resistentes (TWAP/Chainlink), no usar spot price, snapshots en governance. En el lado de proveer flash loans: validar el callback. → Proyecto 21 (flash de Uniswap V3, con validación correcta del callback vía factory).

SC08 · Integer Overflow / Underflow

Desde Solidity 0.8 hay checks nativos (revierte automáticamente). unchecked{} los desactiva (úsalo solo cuando probás que no puede overflowear, para ahorrar gas). En assembly no hay ningún check → reimplementarlos a mano. → Proyecto 20 (safeAdd/safeMul en Yul).

SC09 · Insecure Randomness

Nunca uses esto como random block.timestamp, blockhash, block.prevrandao → manipulables/predecibles por validadores. Solución: Chainlink VRF (random verificable off-chain con prueba). → Proyecto 18.
Bug real que arreglaste (Proyecto 18) El callback de VRF (fulfillRandomWords) empujaba los premios (push payment). Si un ganador era un contrato que rechaza ETH, el callback revertía → la ronda quedaba trabada para siempre y nadie cobraba. Fix: pull pattern (ledger s_prizes + claimPrize()) + ReentrancyGuard. Regla: un callback (VRF/flash) jamás debe poder revertir por algo que controla un tercero.

SC10 · Denial of Service (DoS) & Griefing

Hacer que una función crítica sea inusable: loops no acotados sobre arrays que crecen, depender de un .call a una address que revierte, gas griefing. Prevención: pull over push, evitar loops sobre estructuras ilimitadas, no depender de transfers externos para avanzar estado.

Extra · Signature Replay & malleability (firmas)

Replay: reusar una firma válida en otro contexto. Las 3 defensas (memorizalas): nonce (anti-replay, incrementa con cada uso), deadline (expiración), domain separator de EIP-712 (ata la firma a este contrato + esta chain, anti cross-contract/cross-chain replay).

¿Qué es la signature malleability? Una firma ECDSA son 3 números (r, s, v). La matemática permite que para una firma válida exista una segunda versión igual de válida (flipeando s). Si usás la firma como ID único para anti-replay, el atacante genera la variante (firma lo mismo) y la pasa como "otra" → bypassea el control. Solución: ECDSA.recover de OZ rechaza la s "alta" y address(0) — nunca uses ecrecover crudo. Y mejor anti-replay por nonce, no por hash de la firma. → Proyectos 19, 16; resources/signatures.

Extra · Front-running / MEV

El mempool es público: un bot ve tu tx y la adelanta. Sandwich en swaps. Prevención: amountOutMin (slippage) y deadline en swaps; commit-reveal cuando aplica. → Proyectos 09, 10, 21. También: front-running de permit anotado en el 19.

Extra · delegatecall & proxies

delegatecall ejecuta el código de otro contrato en el storage del que llama. Es la base de los proxies (upgradeability) pero peligroso: storage collision si el layout no coincide, y funciones selfdestruct/no-protegidas pueden secuestrar el proxy. Ver sección 7.

Checklist rápido del auditor
  • ¿Hay .call/transfer a contratos antes de actualizar estado? → CEI / guard.
  • ¿Funciones admin/init sin protección? → access control.
  • ¿Precios de un DEX spot? → oráculo + staleness + TWAP.
  • ¿Se chequea el retorno de transfers? → SafeERC20.
  • ¿Random con block vars? → VRF.
  • ¿Loops sobre arrays que crecen? → DoS.
  • ¿Firmas sin nonce/deadline/domain? → replay.
  • ¿Decimales/unidades distintas mezcladas? → normalizar.
  • ¿Callbacks que pueden revertir por terceros? → pull pattern.

3 · Conceptos DeFi

AMM — Automated Market Maker

Sin orderbook: un pool de dos tokens y una fórmula. Uniswap V2: x · y = k (producto constante). El precio sale del ratio de reservas; comprar mueve el precio (de ahí el slippage). Impermanent loss: si los precios divergen, ser LP rinde menos que holdear. → Proyectos 09, 10.

Uniswap V3 — liquidez concentrada

El LP elige un rango de precio (tickLower/tickUpper) en vez de 0→∞ → mucho más eficiente en capital. Las posiciones son NFTs (ERC-721) del NonfungiblePositionManager. → Proyecto 21.

sqrtPriceX96 y ticks — en criollo
  • sqrtPriceX96: V3 guarda el precio como su raíz cuadrada, en formato fixed-point Q64.96 (= el número × 296, porque Solidity no tiene decimales → para representar decimales multiplicás por 296 y trabajás con enteros). ¿Por qué la raíz? La matemática de liquidez concentrada (cuánto token hay en un rango) queda mucho más simple y precisa con sqrt(price) que con price. Fórmula: price = (sqrtPriceX96 / 2^96)² (token1 por token0).
  • Ticks: discretizan el precio: price = 1.0001^tick → cada tick = 0.01% de precio. Un LP provee entre tickLower y tickUpper.
Para una entrevista no hace falta calcularlo: alcanza con saber QUÉ es (precio como sqrt en fixed-point para que la math de liquidez concentrada sea eficiente) y que los ticks discretizan el precio.

Lending / Borrowing

Sobre-colateralización: pedís prestado menos de lo que vale tu garantía. Health factor / ratio: (colateral_usd × collateralFactor) / deuda_usd; si cae bajo el umbral, sos liquidable. Liquidación: un tercero repaga parte de tu deuda y se lleva colateral + penalidad → mantiene al protocolo solvente. → Proyecto 16.

Staking / Yield Farming — patrón acumulador

El patrón que te van a preguntar (MasterChef / Synthetix) Para repartir rewards proporcional al stake y al tiempo sin loopear sobre todos los usuarios: un acumulador global rewardPerTokenStored (rewards por token desde el inicio, solo sube) y un rewardDebt por usuario (marca de agua de lo ya cobrado). pending = staked × rewardPerToken − rewardDebt. O(1) por usuario. → Proyecto 14.

Governance

Token de voto → propose → vote (quórum + mayoría) → (timelock) → execute. Snapshot voting: el poder de voto se congela en el bloque de creación de la propuesta (ERC20Votes con checkpoints). → Proyecto 15.

Bug que demostraste (Proyecto 15) Si el voto usa balanceOf en vivo (sin snapshot), alguien vota, transfiere los tokens a otra cuenta y vuelve a votar con los mismos tokens (doble voto). Por eso governance real usa snapshots. El test testDoubleVoteByTransferringTokens lo prueba.

Oráculos

Los contratos no pueden leer datos off-chain solos. Chainlink (push): latestRoundData() devuelve precio (8 decimales típico). TWAP (Uniswap): promedio temporal, resistente a manipulación puntual. Siempre: chequear price > 0 + freshness. → Proyectos 11, 16; resources/oracles.

Flash loans

Préstamo sin colateral devuelto en la misma tx (si no devolvés, todo revierte). Usos legítimos: arbitraje, refinanciación, liquidaciones. Riesgo: amplificador de otros bugs. → Proyecto 21.

4 · Tokens & estándares

ERC-20

Fungible. Funciones clave: transfer, approve + transferFrom (modelo de delegación: el user aprueba, el contrato "chupa" con transferFrom). Decimales: 18 típico, pero USDC=6, WBTC=8 → fuente #1 de bugs. SafeERC20: wrapper de OZ que revierte si el token devuelve false (USDT). → 02, 06, 14, 16.

Allowance & el "approve race" (por qué se aprueba 0 primero) Allowance = cuánto puede gastar un spender de tus tokens (lo seteás con approve). El ataque: querés bajar el allowance de 100 a 50 con approve(spender, 50). Un spender malicioso ve tu tx en el mempool y, antes de que se mine, gasta los 100 viejos (transferFrom(100)); cuando se mina tu approve(50), gasta 50 más → gastó 150 en vez de 50. Workaround histórico: aprobar a 0 primero, confirmar, y recién ahí a 50. Lo moderno: increaseAllowance/decreaseAllowance (cambios relativos) o permit (firma puntual). Es más un clásico de entrevista que algo de uso diario.

ERC-721 (NFT)

No fungible, cada tokenId único. _safeMint (no _mint): valida que el receptor sepa recibir NFTs (onERC721Received), si no se pierden. tokenURI + metadata off-chain (IPFS) por costo. → 07, 08.

Firmas off-chain, EIP-712 y Permit

¿Qué es una firma off-chain? Firmás un mensaje con tu private key fuera de la blockchain (gratis, sin gas). Después cualquiera lleva esa firma on-chain y el contrato verifica con ECDSA.recover que la firmó tu address. Sirve para autorizar algo sin pagar gas vos — lo paga un relayer. Ej.: "autorizo retirar 100 USDC" lo firmás gratis, y el protocolo (o un bot) lo ejecuta.
  • El problema de firmar un hash crudo: la wallet te muestra un hexadecimal ilegible → firmás a ciegas (peligroso).
  • EIP-712 (typed data): estándar para firmar datos estructurados legibles — la wallet muestra "Withdraw 100 USDC to 0xabc…", no un hash. El digest final = keccak256("\x19\x01" ‖ domainSeparator ‖ structHash), donde el domain separator (name, version, chainId, address del contrato) ata la firma a ESTE contrato y ESTA chain (anti replay cross-contract/cross-chain).
  • Permit (EIP-2612): aplicar esto al approve. En vez de una tx de approve (que cuesta gas), firmás el permiso off-chain y se ejecuta con permit(...) → approve sin gas, y en una sola tx junto al transferFrom.
  • Las 3 defensas siempre: nonce (anti-replay), deadline (expira), domain separator (contexto).
→ Proyecto 19 (PermitToken + GaslessVault), 16 (depositWithSignature); resources/signatures.

5 · Testing & Foundry

CheatcodeQué hace
vm.prank(x) / startPrankLa próxima llamada (o todas) corren como msg.sender == x.
vm.expectRevert(...)La próxima llamada DEBE revertir (string o selector).
vm.warp(t) / vm.roll(n)Mueve block.timestamp / block.number.
vm.deal(x, n)Setea el balance de ETH de x.
vm.expectEmit(...)Verifica que se emita un evento con los args esperados.
vm.sign(pk, digest)Firma off-chain (para EIP-712/permit).

Tipos de test

  • Unit: happy path + cada revert.
  • Fuzz: una función de test que recibe parámetros; Foundry le mete inputs random (256 por default) buscando romper un assert. Sin assert no testea nada; bound()/vm.assume() acotan inputs inválidos.
  • Invariant: una propiedad que SIEMPRE debe valer. Foundry tira secuencias random de llamadas al contrato (funciones + args al azar) y tras cada una chequea el invariante (ej. sumaBalances == totalSupply, o "k nunca baja" en un AMM). Diferencia con fuzz: fuzz prueba una función; invariant prueba el sistema entero.
  • Fork testing: corre contra un fork de mainnet/Arbitrum con contratos reales (Uniswap, Chainlink). Imprescindible para integraciones. → 09, 10, 11, 16, 21.

Fuzz — ejemplo

function testFuzz_deposit(uint256 amount) public {
    amount = bound(amount, 1, 1e24);   // rango válido
    token.mint(user, amount);
    vm.prank(user);
    vault.deposit(amount);
    // debe valer para CUALQUIER amount:
    assertEq(vault.balanceOf(user), amount);
}

Invariant — ejemplo

// Foundry llama funciones random del "handler"
// y tras cada secuencia chequea esto:
function invariant_totalSupplyMatchesBalances()
    public view
{
    assertEq(token.totalSupply(), handler.sumBalances());
}
Frase que suma en entrevista "El test no está para confirmar que el código anda, sino para intentar romperlo." Y: coverage 100% ≠ correcto — las branches que quedan sin cubrir suelen ser guardas inalcanzables; saber por qué vale más que inflar el número.
Mocks Y fork (no es uno u otro) Fork = integración con precios/tokens reales. Mocks = control (ej. MockAggregator para bajar el precio y forzar una liquidación, cosa que no podés hacer con un feed real). → Proyecto 16.

6 · Optimización de gas

  • Storage packing: agrupar variables chicas en un slot (2×uint128, uint96+address).
  • constant / immutable en vez de storage para valores fijos.
  • Custom errors en vez de require con string.
  • calldata en vez de memory para args de funciones external.
  • Cachear storage en memory: leer una state var en un loop una sola vez.
  • unchecked{} donde está probado que no overflowea (ej. i++ en loops).
  • Eventos en vez de storage para data que solo se lee off-chain.
  • via_ir = true + optimizer para resolver "stack too deep" sin deformar el código. → 10.
"Stack too deep" y via_ir — qué es y cuándo aparece El EVM tiene un stack donde solo podés acceder a las ~16 variables más recientes. Si una función tiene demasiadas variables locales + parámetros + returns "vivas" al mismo tiempo (>16), el compilador no las puede ubicar → error "stack too deep". Aparece en funciones con muchas variables — ej. en el proyecto 10, las de add/remove liquidity (varios amountA/amountB/min/deadline/returns a la vez). via_ir = true compila vía Yul IR, que reusa slots de forma más inteligente y lo resuelve sin reescribir el código (la alternativa manual sería agrupar variables en struct o scoparlas en bloques { }). Trade-off: compila más lento.

7 · Upgradeability (proxies)

Los contratos son inmutables; para "actualizarlos" se separa lógica de estado con un proxy que hace delegatecall a una implementación. → resources/upgradeability (Transparent, UUPS, Beacon).

PatrónIdea
TransparentEl proxy decide: admin → funciones de upgrade; users → la lógica. Evita colisiones de selector.
UUPSLa función de upgrade vive en la implementación (más barato de deployar). Hay que protegerla sí o sí.
BeaconMuchos proxies apuntan a un beacon; actualizás uno y se actualizan todos.
¿Dónde vive el address de la implementación? En un slot de storage fijo del PROXY, definido por EIP-1967 para que ninguna variable normal lo pise: keccak256("eip1967.proxy.implementation") - 1 (un slot "raro" a propósito). El fallback del proxy lee ese slot y hace delegatecall a esa address → ejecuta la lógica sobre el storage del proxy. Actualizar = escribir una address nueva en ese slot (en Transparent lo hace el ProxyAdmin; en UUPS la propia implementación, protegida por _authorizeUpgrade). El admin/owner vive en otro slot EIP-1967 (...proxy.admin). Por eso el storage layout importa: la lógica escribe en el storage del proxy; si reordenás variables entre versiones, pisás datos.
Riesgos clave
  • Storage collision: el layout del proxy y la implementación deben coincidir; al actualizar, solo agregar variables al final (nunca reordenar/insertar).
  • Initializer en vez de constructor: el constructor corre en el contexto de la implementación, no del proxy → usar initialize() con initializer (y proteger contra doble init).
  • UUPS sin protección de _authorizeUpgrade = cualquiera te cambia la lógica.

9 · Banco de preguntas

Auto-testeate: leé la pregunta, respondela en voz alta, después abrí la respuesta.

¿Qué es reentrancy y cómo se previene?▶

Una llamada externa devuelve el control al atacante antes de actualizar el estado, permitiéndole re-entrar y repetir la operación con estado viejo. Defensas: CEI (estado antes de la llamada), ReentrancyGuard, pull over push.

storage vs memory vs calldata▶

storage: persistente, caro. memory: temporal, mutable. calldata: read-only, args de funciones external, la más barata.

msg.sender vs tx.origin — ¿cuál para auth?▶

msg.sender (caller directo). tx.origin es el EOA original y es vulnerable a phishing → nunca para auth.

transfer / send / call para enviar ETH▶

transfer/send tienen límite de 2300 gas (rompe con contratos que tienen receive con lógica) → deprecados. .call{value:}("") es el estándar; chequear el bool success y aplicar CEI/guard.

constant vs immutable▶

constant: compile-time, inlineado. immutable: se setea en el constructor, fijo después. Ambos baratos vs storage.

¿Cómo conseguís aleatoriedad on-chain?▶

NO con block.timestamp/blockhash/prevrandao (manipulables). Con Chainlink VRF (random verificable, flujo async request → fulfill).

¿Cómo se previene la manipulación de oráculos?▶

No usar spot price de un DEX; usar Chainlink / TWAP; validar price > 0 y updatedAt (staleness). Cuidado con flash loans que mueven pools de baja liquidez.

¿Qué es un flash loan y por qué es peligroso?▶

Préstamo sin colateral devuelto en la misma tx. No es el bug en sí: es el amplificador que da capital "infinito" para explotar oracle manipulation o lógica débil.

approve / transferFrom — ¿cómo funciona?▶

El user llama approve(spender, monto); el spender después usa transferFrom(user, to, monto). Es el consentimiento explícito del ERC-20. Usar SafeERC20 por los tokens que devuelven false.

¿Qué es EIP-712 y para qué sirve el domain separator?▶

Firmas de typed data legibles. El domain separator (name, version, chainId, contract address) ata la firma a este contrato y esta chain → previene replay cross-contract/cross-chain. Sumá nonce (anti-replay) y deadline (expiración).

¿Por qué Solidity 0.8 no necesita SafeMath?▶

Tiene checks de overflow/underflow nativos que revierten. unchecked{} los desactiva para ahorrar gas donde es seguro.

¿Cómo funciona un proxy upgradeable?▶

Un proxy guarda el estado y hace delegatecall a una implementación (lógica). Para actualizar, cambiás la dirección de implementación. Riesgos: storage collision (solo agregar vars al final) y usar initialize() en vez de constructor.

¿Qué es el slippage y cómo se protege?▶

La diferencia entre el precio esperado y el real al ejecutar un swap (por movimiento del pool / front-running). Se protege con amountOutMin y deadline.

view vs pure▶

view: lee estado, no lo modifica. pure: ni lee ni modifica. Llamadas externas a ambas son gratis (eth_call).

Patrón acumulador de rewards (yield farming)▶

Acumulador global rewardPerToken + rewardDebt por usuario. pending = staked × rewardPerToken − rewardDebt. Reparte en O(1) sin loopear usuarios.

¿Cómo prevenís el doble voto en governance?▶

Con snapshots: congelar el poder de voto en el bloque de la propuesta (ERC20Votes + checkpoints). Si usás balanceOf en vivo, alguien transfiere y revota con los mismos tokens.

¿Qué es la EVM y qué es el gas?▶

EVM (Ethereum Virtual Machine): la "computadora" descentralizada que ejecuta el bytecode de los contratos, igual en todos los nodos. Gas: la unidad que mide cuánto cómputo/almacenamiento consume una operación; pagás gas usado × gas price en ETH. Evita loops infinitos y spam (todo cuesta).

¿Qué es un AMM?▶

Automated Market Maker: un exchange sin orderbook. Un pool con dos tokens y una fórmula (x·y=k en V2) define el precio según el ratio de reservas. Comprar mueve el precio (slippage). Los LPs aportan liquidez y cobran fees.

¿Qué es upgradeability y dónde vive el address de la implementación?▶

Los contratos son inmutables; para "actualizarlos" un proxy guarda el estado y hace delegatecall a una implementación (la lógica). El address de la implementación vive en un slot fijo del proxy definido por EIP-1967 (keccak256("eip1967.proxy.implementation")-1). Actualizar = escribir un address nuevo ahí (UUPS: desde la implementación con _authorizeUpgrade; Transparent: desde el ProxyAdmin).

¿Qué es una firma off-chain y para qué sirve?▶

Firmar un mensaje con tu private key fuera de la chain (gratis). Otro lo lleva on-chain y el contrato verifica con ECDSA.recover que lo firmaste. Sirve para autorizar acciones sin pagar gas vos (lo paga un relayer). EIP-712 lo hace legible (typed data); permit aplica esto al approve.

¿Qué es la signature malleability?▶

Para una firma ECDSA válida existe una "segunda versión" igual de válida (flipeando s). Si usás la firma como ID anti-replay, el atacante pasa la variante como "otra" y la reusa. Fix: ECDSA.recover de OZ (low-s, rechaza address(0)); anti-replay por nonce.

TWAP vs staleness▶

TWAP: precio promedio ponderado por tiempo (ej. 30 min, oráculo de Uniswap); resiste manipulación puntual. Staleness: chequear que el precio del oráculo no esté viejo (updatedAt reciente + price > 0) antes de usarlo.

¿Qué es "stack too deep" y cómo se resuelve?▶

El EVM solo accede a las ~16 variables más recientes del stack. Si una función tiene >16 variables vivas a la vez, el compilador falla con "stack too deep". Se resuelve con via_ir = true (compila vía Yul IR, reusa slots) o agrupando variables en structs/bloques.

fuzz vs invariant test▶

Fuzz: una función con parámetros que Foundry llama con inputs random buscando romper un assert. Invariant: una propiedad global que siempre debe valer, verificada tras secuencias random de llamadas a todo el contrato.

¿Por qué se aprueba 0 antes de cambiar un allowance?▶

Por el approve race: bajar de 100 a 50 deja una ventana donde el spender malicioso gasta los 100 viejos y después los 50 nuevos (150 total). Aprobar a 0 primero (o usar increase/decreaseAllowance o permit) lo mitiga.
